API Reference

Complete guide to both endpoints with real examples.

Authentication​

All API requests require the X-Docflow-Gateway-Secret header with a valid gateway token.

Important: If you're using the RapidAPI marketplace, authentication is handled automatically—just use your RapidAPI API key.

Endpoint 1: POST /process​

What it does: Process a document and get structured chunks as JSON.

Base URL (Direct Backend):

https://docflow-xvmh.onrender.com/process

Recommended: Use via RapidAPI

https://docflow.p.rapidapi.com/process

Parameters​

You send ONE of these (file OR text):

NameTypeRequiredLimitsDescription
fileFile UploadNoMax 10MBPDF, DOCX, or TXT file
textStringNo—Raw text input (if no file)

Plus these optional tuning parameters:

NameTypeDefaultRangeDescription
chunk_sizeInteger2000200–10000Characters per chunk
overlapInteger2000–5000Character overlap between chunks
modeStringbasicbasic, semantic-liteChunking algorithm

Example 1: Upload a PDF (Direct Backend)​

curl -X POST https://docflow-xvmh.onrender.com/process \
  -H "X-Docflow-Gateway-Secret: YOUR_GATEWAY_SECRET" \
  -F "file=@research_paper.pdf" \
  -F "chunk_size=1500" \
  -F "overlap=100" \
  -F "mode=semantic-lite"

Example 1b: Upload a PDF (via RapidAPI - Recommended)​

curl -X POST https://docflow.p.rapidapi.com/process \
  -H "x-rapidapi-key: YOUR_RAPIDAPI_KEY" \
  -H "x-rapidapi-host: docflow.p.rapidapi.com" \
  -F "file=@research_paper.pdf" \
  -F "chunk_size=1500" \
  -F "overlap=100" \
  -F "mode=semantic-lite"

Response:

{
  "request_id": "req_pdf_001",
  "document": {
    "filename": "research_paper.pdf",
    "file_type": "pdf",
    "total_characters": 48000,
    "num_chunks": 18
  },
  "result": {
    "chunks": [
      {
        "chunk_id": 0,
        "text": "Abstract: This study investigates...",
        "title": "Abstract",
        "section_label": "Preamble",
        "tokens_estimate": 180,
        "char_range": [0, 1500]
      },
      {
        "chunk_id": 1,
        "text": "Introduction: The problem of inefficient...",
        "title": "Introduction",
        "section_label": "Main Body",
        "tokens_estimate": 420,
        "char_range": [1400, 2900]
      }
    ]
  },
  "metadata_summary": {
    "total_chunks": 18,
    "total_tokens_estimate": 6240
  }
}

What to use:

  • chunks[].text — The actual content (use for RAG)
  • chunks[].title — What this chunk is about
  • chunks[].tokens_estimate — Cost to send this to an LLM
  • total_tokens_estimate — Total cost if you send all chunks

Example 2: Send Raw Text with Custom Chunk Size​

curl -X POST https://docflow-xvmh.onrender.com/process \
  -F "text=Customer feedback: The product is great but slow. Pricing is competitive. Team is responsive. Would recommend." \
  -F "chunk_size=100" \
  -F "mode=basic"

Response:

{
  "request_id": "req_txt_002",
  "document": {
    "filename": "input.txt",
    "file_type": "txt",
    "total_characters": 150,
    "num_chunks": 2
  },
  "result": {
    "chunks": [
      {
        "chunk_id": 0,
        "text": "Customer feedback: The product is great but slow.",
        "tokens_estimate": 12
      },
      {
        "chunk_id": 1,
        "text": "Pricing is competitive. Team is responsive. Would",
        "tokens_estimate": 11
      }
    ]
  },
  "metadata_summary": {
    "total_chunks": 2,
    "total_tokens_estimate": 23
  }
}

Chunking Modes Explained​

mode=basic (Default)

  • Fixed-size character windows
  • Simple, predictable
  • Splits mid-sentence sometimes (doesn't care about structure)
  • Best for: uniform content, speed

mode=semantic-lite

  • Respects paragraph and sentence boundaries
  • Adds intelligent title and section_label
  • Slower (2–3 seconds more)
  • Best for: complex documents, quality over speed

Endpoint 2: POST /pack​

What it does: Same as /process but exports everything to a ZIP file.

Base URL:

https://docflow-xvmh.onrender.com/pack

Parameters​

Same as /process, plus:

NameTypeDefaultDescription
zip_nameStringpack.zipWhat to name the output ZIP

Example: Export to ZIP​

curl -X POST https://docflow-xvmh.onrender.com/pack \
  -F "file=@document.docx" \
  -F "chunk_size=2000" \
  -F "zip_name=my_document.zip" \
  -o my_document.zip

ZIP contains 3 files:

  1. results.json — Full response (same as /process)
  2. document.md — Content as clean markdown
  3. chunks.jsonl — One JSON object per line (easier for big data)

Use cases:

  • Archive processed documents
  • Import into data pipelines
  • Batch processing scripts

Error Codes​

If something goes wrong, you get:

{
  "detail": {
    "code": "ERROR_CODE",
    "message": "Human readable message"
  }
}

Common errors:

CodeStatusWhat Happened
MISSING_INPUT400You didn't send file or text
FILE_TOO_LARGE413File bigger than 10MB
UNSUPPORTED_TYPE415File type not PDF/DOCX/TXT
INVALID_CHUNK_SIZE400chunk_size outside 200–10000 range
INVALID_MODE400mode is not basic or semantic-lite
TOO_MANY_CHUNKS400Would create more than 100 chunks (lower chunk_size)
PROCESSING_TIMEOUT504Request took >10 seconds (try smaller file or larger chunk_size)

Rate Limits & Quotas​

Depends on your plan:

PlanRequests/MonthPer-Second Limit
Free502
Starter5005
Pro200010

Hit the limit? Requests return 429 (Too Many Requests). Upgrade your plan.

Best Practices​

  1. Start with mode=basic — Fast, no surprises
  2. Use chunk_size=2000 — Good default for most LLMs
  3. Check tokens_estimate — Know your costs before feeding to LLM
  4. Batch requests — Don't send 100 files one-at-a-time
  5. Store request_id — Useful for debugging
