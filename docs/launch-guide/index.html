Launch Guide

Use this page as the one-step-at-a-time path for getting the project ready to publish, then keeping it updated after launch.

1. Decide the launch shape​

Before touching anything else, decide which of these is your public surface:

  • Docusaurus docs as the main product site
  • Direct backend access for your own app or internal users
  • RapidAPI as the public marketplace entry point
  • Postman as the manual QA and demo surface

For this project, the cleanest setup is all four together:

  1. Docusaurus explains the product.
  2. Postman proves the API works.
  3. RapidAPI handles marketplace distribution.
  4. The backend serves the actual requests.

2. Keep the backend as the source of truth​

The docs should describe the backend, not the other way around.

Do this first:

  1. Start the backend locally.
  2. Confirm GET /health works.
  3. Confirm POST /process works with a plain text request.
  4. Confirm POST /v2/process works with at least one sample schema.
  5. Confirm POST /v2/batch, POST /v2/index, and GET /v2/search behave as expected.
  6. Confirm POST /v2/jobs and GET /v2/jobs/{job_id} work end to end.

If any endpoint changes, update the docs and the Postman collection before publishing anything else.

3. Prepare the docs site​

The Docusaurus site is where people should understand the project quickly.

Update these pages in order:

  1. intro - short product summary and quick links.
  2. getting-started - local setup and first request.
  3. launch-guide - this page, the full workflow.
  4. api and api-reference - exact request/response contract.
  5. postman - how to import and use the collection.
  6. rapidapi - marketplace setup and request headers.
  7. launch-checklist - the final verification list.

Use the docs site to keep the wording simple and concrete. If something is confusing in the docs, it is usually a signal that the launch workflow is still too vague.

4. Publish or update Docusaurus​

When the docs look correct locally, publish them.

Suggested release flow:

  1. Build locally with npm run build inside docflow-docs.
  2. Fix any build errors before publishing.
  3. Deploy the docs site to whatever hosting you choose.
  4. Open the live site and test the top navigation, sidebar, and download links.

Things to verify on the live site:

  • The homepage points to the launch guide.
  • The Postman download links work.
  • The RapidAPI page matches the current marketplace product.
  • The launch checklist is easy to find.

5. Set your backend environment variables​

Decide which features are on for launch and set the backend variables accordingly.

Minimum useful set:

  • DOCFLOW_GATEWAY_SECRET if you want gateway protection.
  • DOCFLOW_API_KEY_SECRET if you want signed API keys.
  • DOCFLOW_API_KEY_ADMIN_SECRET if you want to issue keys.
  • DOCFLOW_REQUIRE_API_KEY=true if API keys should be required.
  • DOCFLOW_WEBHOOK_SECRET if you want signed webhook payloads.
  • DOCFLOW_ALLOW_EXTERNAL_DELIVERY=true only when you are ready for real webhook delivery.

Recommended safer starting point:

  1. Keep DOCFLOW_REQUIRE_API_KEY=false while you are testing.
  2. Keep DOCFLOW_ALLOW_EXTERNAL_DELIVERY=false until webhook behavior is verified.
  3. Add secrets one at a time so you can tell which protection layer is affecting a request.

6. Import and test Postman​

Postman is your repeatable QA harness.

Do this:

  1. Import the collection.
  2. Import the environment.
  3. Set base_url to your local backend first.
  4. Run GET /health.
  5. Run POST /v2/process.
  6. Run POST /v2/batch.
  7. Run POST /v2/admin/api-keys if key issuance is enabled.
  8. Run POST /v2/jobs and then poll GET /v2/jobs/{job_id}.

Use Postman to answer one question only: does the documented request still work exactly the way the backend expects?

7. Configure RapidAPI​

RapidAPI should mirror the backend, not invent a new contract.

Set it up like this:

  1. Create or update the API product.
  2. Point the product at the deployed backend base URL.
  3. Add the public routes you want to expose.
  4. Make sure the request headers shown in the docs are configured.
  5. Add one test request for POST /v2/process.
  6. Publish or republish the API product.

When you test it, check all of these:

  • x-rapidapi-key is accepted.
  • x-rapidapi-host matches the product host.
  • The backend still accepts the request body format.
  • The response returned through RapidAPI matches the direct backend response.

If you use a gateway secret behind RapidAPI, document that clearly and keep it out of the public examples.

8. Do a final launch pass​

Before you say the project is live, run the final checklist in this order:

  1. Docs site is live and readable.
  2. Backend health check passes.
  3. Postman collection works.
  4. RapidAPI request works.
  5. API keys issue correctly if enabled.
  6. Jobs complete and can be polled.
  7. Search indexing and retrieval work.
  8. Webhook delivery works or is intentionally disabled.

If any item fails, fix the backend or the docs before shipping.

9. Keep it updated after launch​

After release, treat the docs as a living contract.

Whenever you change something, update these three things together:

  1. Backend route or behavior.
  2. Docusaurus docs.
  3. Postman collection and RapidAPI setup.

That keeps the project from drifting into three slightly different versions of the same API.

10. Good next moves after launch​

Once the first public version is out, the most useful follow-ups are:

  1. Add monitoring and uptime alerts.
  2. Add persistent storage for search/indexing if needed.
  3. Add usage tracking or rate limiting if traffic grows.
  4. Tighten examples and onboarding based on what users actually ask about.
  5. Add a changelog page so future updates are easier to communicate.
