Parseflow Launch Step-by-Step Guide

This is the single guide to finish the public launch.

It covers:

  • Azure setup
  • admin secret
  • demo API key
  • Postman public collection
  • key tracking
  • publishing checklist

1) Gather Azure info​

You need 3 values:

  • Subscription ID
  • Resource Group
  • Container App name

How to find them​

Subscription ID

  • Azure Portal → Subscriptions → copy Subscription ID
  • Or run:
az account list --output table

Resource Group

  • Azure Portal → Resource groups → copy the group name
  • Or run:
az group list --output table

Container App name

  • Azure Portal → Container Apps → copy the app name
  • Or run:
az containerapp list --resource-group "<RESOURCE_GROUP>" --output table

2) Generate an admin secret​

Run:

openssl rand -hex 32

Save the result somewhere private.

Do not put it in Postman public docs or GitHub.

3) Set the admin secret in Azure​

Use your real values:

az containerapp update \
  --name "<CONTAINER_APP_NAME>" \
  --resource-group "<RESOURCE_GROUP>" \
  --set-env-vars DOCFLOW_API_KEY_ADMIN_SECRET="<ADMIN_SECRET>"

Then wait about 30 seconds.

Verify health:

curl -sS "https://<APP_FQDN>/health" | jq .

Expected:

{"status":"ok","version":"0.2.0"}

4) Create a demo API key​

Use the admin endpoint:

curl -sS -X POST "https://<APP_FQDN>/v2/admin/api-keys" \
  -H "Content-Type: application/json" \
  -H "X-Docflow-Admin-Secret: <ADMIN_SECRET>" \
  -d '{
    "name": "public-demo",
    "scopes": ["process"],
    "expires_in_days": 0.04167,
    "rate_limit_per_hour": 100
  }' | jq .

This creates a short-lived demo key.

Save the returned api_key.

Recommended limits:

  • 1 hour TTL
  • 100 requests/hour
  • only process scope

5) Test the API​

Health​

curl -sS "https://<APP_FQDN>/health" | jq .

File upload​

curl -sS -X POST "https://<APP_FQDN>/v2/process" \
  -H "X-API-Key: <DEMO_API_KEY>" \
  -F "file=@sample.pdf" \
  -F "text=Invoice total $125 due Friday" \
  --output parseflow-output.zip

Text-only​

curl -sS -X POST "https://<APP_FQDN>/v2/process" \
  -H "X-API-Key: <DEMO_API_KEY>" \
  -F "text=Invoice total $125 due Friday"

Key info​

curl -sS "https://<APP_FQDN>/v2/me" \
  -H "X-API-Key: <DEMO_API_KEY>" | jq .

Usage info​

curl -sS "https://<APP_FQDN>/v2/usage" \
  -H "X-API-Key: <DEMO_API_KEY>" | jq .

6) Set up Postman​

Import:

  • parseflow-demo-public.postman_collection.json

Create a Postman environment with:

  • baseUrl = https://<APP_FQDN>
  • API_KEY = your demo key
  • API_KEY_HEADER_NAME = X-API-Key

Do not add admin secrets to the public environment.

Public requests in the collection​

  • Health
  • Process (file upload)
  • Process (text-only)
  • Key Info
  • Usage Info

Important​

The text-only request must use form-data with a text field. It is not raw JSON.

7) Publish Postman publicly​

In Postman:

  1. Open the collection
  2. Publish docs or share the public workspace
  3. Make sure only public requests are included
  4. Keep admin requests private

What people should see publicly:

  • health check
  • file upload demo
  • text-only demo
  • key info
  • usage info

What should stay private:

  • admin secret
  • admin key issuance details
  • any operator-only requests

8) Add a simple link on your site/docs​

You do not need a separate website flow. Just add a link like:

  • “Try the API in Postman”
  • “Public demo available in Postman”

That is enough.

9) Track keys​

For users​

They can check:

  • /v2/me
  • /v2/usage

For you as operator​

Track these fields when you issue a key:

  • key_id
  • label
  • issued time
  • expiry time
  • quota/rate limit

Best practice:

  • keep a simple spreadsheet or note
  • rotate demo keys often
  • revoke if abused

If you want stronger tracking later, add an admin list/revoke dashboard.

10) Final launch checklist​

Before you call it done:

  • admin secret set in Azure
  • health check works
  • file upload works
  • text-only works
  • demo key works
  • /v2/me works
  • /v2/usage works
  • Postman collection imports
  • Postman docs are public
  • admin requests are not public
  • README has a Postman link
  • demo key is short-lived
  • demo key is rate-limited
  • you know how to rotate/revoke keys

11) What is left after launch​

After everything above is done, the remaining work is only maintenance:

  • rotate demo keys
  • monitor usage
  • revoke bad keys
  • update docs if the API changes
