Publishing Checklist for Parseflow API

Use this checklist to verify everything is ready before announcing the public API and demo.

✅ Pre-Publishing Checklist​

Admin Secret & Security​

  • Admin secret generated via openssl rand -hex 32
  • Admin secret stored in DOCFLOW_API_KEY_ADMIN_SECRET (Azure env var)
  • Admin secret is NOT in git, public docs, or Postman collection
  • Container App restarted after setting env var (wait 30+ seconds)
  • Admin endpoints (/v2/admin/*) return 401 without correct header

Demo API Key​

  • Demo key created via POST /v2/admin/api-keys with admin secret
  • Demo key has short TTL (1 hour recommended, max 7 days)
  • Demo key has rate limit (100 requests/hour recommended)
  • Demo key has limited scopes (only "process")
  • Demo key works with sample requests (health, process)

Deployment & Health​

  • GET /health returns {"status":"ok","version":"0.2.0"}
  • POST /v2/process accepts form-data (file + analyze)
  • Process endpoint returns ZIP or JSON with proper headers
  • Demo key works with at least one process test (text-only or small file)
  • App is publicly accessible (HTTPS, no 403/timeout)

Documentation​

  • Postman Setup Guide created and reviewed
  • Azure Setup Guide created for operators
  • parseflow-demo-public.postman_collection.json exported and tested
  • README updated with links to Postman guide and demo info
  • No admin secrets or operator-only info in public docs

Postman Collection​

  • Public folder contains: Health, Process (file), Process (text)
  • No "Operator" folder in public collection export
  • All requests use {{baseUrl}} and {{API_KEY}} variables
  • Tests written for each request (status checks, response validation)
  • Collection can be imported successfully and requests run without errors

Monitoring & Safety​

  • Application Insights enabled (or equivalent logging)
  • Rate-limiting configured on demo key
  • Plan for key rotation documented (e.g., daily)
  • Abuse response plan documented (how to revoke compromised key)
  • Support contact info added to Postman docs

Repo & Release​

  • README updated with public demo section
  • parseflow-demo-public.postman_collection.json added to repo root
  • Release notes prepared (link to Postman docs, version, features)
  • Git tag created (e.g., v0.2.0-public-demo)
  • All changes committed and pushed to main branch

📋 Final Verification Steps​

Before announcing publicly, run these tests:

Test 1: Health Check​

curl -sS "https://your-api-url/health" | jq .
# Expected: {"status":"ok","version":"0.2.0"}

Test 2: Process with Demo Key​

DEMO_KEY="your-demo-key"
API_URL="https://your-api-url"

curl -sS -X POST "$API_URL/v2/process" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $DEMO_KEY" \
  -d '{"text": "Invoice total $125 due Friday"}' | head -c 100
# Expected: 200 status, JSON or ZIP response

Test 3: Admin Endpoint is Protected​

curl -sS -X POST "https://your-api-url/v2/admin/api-keys" \
  -H "Content-Type: application/json" \
  -H "X-Docflow-Admin-Secret: wrong-secret" \
  -d '{"name":"test"}' | jq .
# Expected: 401 UNAUTHORIZED

Test 4: Postman Collection Imports​

  1. Open Postman
  2. File → Import → Select parseflow-demo-public.postman_collection.json
  3. Confirm 3 requests imported under "Public" folder
  4. Set environment variables (baseUrl, API_KEY)
  5. Run "Health" request → expect 200 ✅
  6. Run "Process (text-only)" → expect 200 ✅

🚀 Publishing Steps (Once Checklist Passes)​

Step 1: Create GitHub Release​

git tag -a v0.2.0-public-demo -m "Public demo release with Postman collection"
git push --tags

Add release body to GitHub:

  • Links to Postman docs
  • Demo API key info (TTL, quota)
  • Instructions for importing collection
  • Caution about operator-only endpoints

Step 2: Announce on Social Media​

Twitter/X:

🎉 Parseflow API is now public! 

Parse PDFs, DOCX, and text with structured output.

📥 Import Postman collection: [link]
⚡ Demo key: pk_demo_... (100 req/hr, 1-hour TTL)
📖 Docs: https://docs.parseflow.tech

Try it now! #API #DocumentParsing #OpenSource

LinkedIn, Dev.to, Product Hunt (similar messaging)

Step 3: Update Product Listings​

  • Whop product page: link to Postman docs
  • Product Hunt: link to GitHub, Postman docs
  • Dev.to post: code examples + demo key

Step 4: Monitor Usage​

  • Check Application Insights for errors
  • Monitor demo key quota usage (should see < 100 req/hr until promotion)
  • Prepare to rotate demo key daily

⚠️ If Something Goes Wrong​

Demo key stopped working​

  1. Verify it hasn't expired (1-hour TTL)
  2. Check if rate limit was hit (100 req/hour)
  3. Create a new key using admin endpoint (see Azure Setup Guide)
  4. Update Postman and docs with new key

Admin secret leaked​

  1. Generate new admin secret: openssl rand -hex 32
  2. Update Azure: az containerapp update --set-env-vars DOCFLOW_API_KEY_ADMIN_SECRET="new-secret"
  3. Wait 30 seconds for restart
  4. Create new demo key with new admin secret
  5. Revoke all old keys (if supported)
  6. Update docs with new key

High error rate or abuse​

  1. Check Application Insights logs
  2. Revoke demo key immediately
  3. Create new key with lower quota or shorter TTL
  4. Announce maintenance/limited access on social media

📞 Support Handoff​

Once published, document:

For users:

  • Email: [your-email]
  • GitHub Issues: [your-repo]
  • Expected response time: 24 hours

For operators:

  • Admin secret stored securely (1Password, etc.)
  • Key rotation schedule: daily
  • Monitoring dashboard: Azure Portal → Application Insights

Final Notes​

✅ Estimated time to complete: 2-3 hours ✅ No Azure resource creation needed (uses existing app) ✅ Safe to test multiple times (non-breaking) ✅ Can revoke/rotate demo key without downtime

When in doubt, follow the Azure Setup Guide step-by-step.

Good luck! 🚀
